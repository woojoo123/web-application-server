# 실습을 위한 개발 환경 세팅
* https://github.com/slipp/web-application-server 프로젝트를 자신의 계정으로 Fork한다. Github 우측 상단의 Fork 버튼을 클릭하면 자신의 계정으로 Fork된다.
* Fork한 프로젝트를 eclipse 또는 터미널에서 clone 한다.
* Fork한 프로젝트를 eclipse로 import한 후에 Maven 빌드 도구를 활용해 eclipse 프로젝트로 변환한다.(mvn eclipse:clean eclipse:eclipse)
* 빌드가 성공하면 반드시 refresh(fn + f5)를 실행해야 한다.

# 웹 서버 시작 및 테스트
* webserver.WebServer 는 사용자의 요청을 받아 RequestHandler에 작업을 위임하는 클래스이다.
* 사용자 요청에 대한 모든 처리는 RequestHandler 클래스의 run() 메서드가 담당한다.
* WebServer를 실행한 후 브라우저에서 http://localhost:8080으로 접속해 "Hello World" 메시지가 출력되는지 확인한다.

# 각 요구사항별 학습 내용 정리
* 구현 단계에서는 각 요구사항을 구현하는데 집중한다. 
* 구현을 완료한 후 구현 과정에서 새롭게 알게된 내용, 궁금한 내용을 기록한다.
* 각 요구사항을 구현하는 것이 중요한 것이 아니라 구현 과정을 통해 학습한 내용을 인식하는 것이 배움에 중요하다. 

### 요구사항 1 - http://localhost:8080/index.html로 접속시 응답
#### 새로 알게 된 내용
* `connection.getInputStream()`은 해당 소켓으로 들어오는 데이터를 읽기 위한 `InputStream`을 반환한다. 즉, 브라우저가 서버로 보낸 요청 데이터를 이 스트림을 통해 읽을 수 있다.
* `InputStream`은 바이트 단위로 데이터를 읽기 때문에, 텍스트 형태의 HTTP 요청을 처리하기 위해 `InputStreamReader`로 감싸 문자 스트림으로 변환할 수 있다.
* `BufferedReader`는 문자 스트림을 버퍼링해 더 효율적으로 읽을 수 있게 해 주고, `readLine()`을 통해 HTTP 요청을 한 줄씩 편하게 처리할 수 있다.
* HTTP 요청의 첫 줄인 Request Line에서 URI를 추출한 뒤, 해당 URI에 맞는 파일을 찾아 응답할 수 있다.
* 응답을 보낼 때는 헤더에 `Content-Length`로 파일 길이를 내려주고, 바디에는 실제 파일 내용을 전달해야 한다.
* `Files.readAllBytes()`는 파일 전체를 한 번에 읽어 `byte[]`로 반환하는 메서드이며, 매개변수로는 읽어 올 파일의 경로를 전달한다.
* 브라우저에서 `/index.html`을 한 번 요청하더라도 실제로는 여러 개의 추가 요청이 발생할 수 있다. 그 이유는 웹 페이지가 HTML만으로 구성되지 않고 CSS, JavaScript, 이미지 같은 여러 정적 자원으로 이루어져 있기 때문이다.
* 서버는 먼저 `index.html`에 대한 응답만 보내고, 브라우저는 그 HTML을 해석한 뒤 필요한 CSS, JavaScript, 이미지 파일을 다시 서버에 요청한다. 따라서 하나의 웹 페이지를 정상적으로 띄우기 위해서는 클라이언트와 서버 사이에 여러 번의 요청과 응답이 오간다.
* 로그를 통해 실제 요청 흐름을 확인하는 습관이 중요하다는 점도 알게 되었다. 어떤 요청이 들어왔는지, 어떤 포트로 연결되었는지, 어떤 응답이 나가는지를 로그로 확인하면 동작을 이해하기 훨씬 쉽다.
* 하나의 페이지를 불러오는 과정에서 여러 요청이 발생하더라도 각 요청은 서로 다른 클라이언트 포트로 연결될 수 있다. 서버는 이 요청들을 순차적으로만 처리하는 것이 아니라, 요청마다 `RequestHandler` 스레드를 생성해 동시에 처리할 수 있다.

#### 궁금한 내용
* `BufferedReader`를 사용하는 이유가 단순히 `readLine()`으로 한 줄씩 읽기 편해서인지, 아니면 성능상 이점도 큰지 더 확인해 보고 싶다.
* 현재는 요청마다 새로운 스레드를 생성하는 방식인데, 요청 수가 많아졌을 때도 이 방식이 적절한지 궁금하다.
* HTML 이외의 CSS, JavaScript, 이미지 파일을 응답할 때 `Content-Type`을 어떤 기준으로 나누어 처리하는지 더 알아보고 싶다.

### 요구사항 2 - get 방식으로 회원가입
#### 새로 알게 된 내용
* 회원가입 폼은 `method="get"`, `action="/user/create"`로 설정되어 있기 때문에, 사용자가 입력한 값은 Request Body가 아니라 URL 뒤의 Query String으로 전달된다.
* 따라서 회원가입 기능을 구현할 때 서버가 실제로 처리해야 하는 요청은 `/user/form.html` 요청이 아니라, 폼 전송 후 발생하는 `/user/create?userId=...&password=...&name=...&email=...` 요청이다.
* GET 요청에서는 HTTP 요청 메시지의 첫 줄(Request Line)이 중요했다. 서버는 첫 줄을 읽은 뒤 공백 기준으로 나누어 메서드, URI, HTTP 버전을 구분할 수 있다.
* URI에 `?`가 포함되어 있으면, `?` 앞은 요청 경로이고 `?` 뒤는 Query String이다. 회원가입 정보는 바로 이 Query String에 담겨 전달된다.
* 이 프로젝트에서는 `HttpRequestUtils.parseQueryString()`을 사용해 `userId`, `password`, `name`, `email` 값을 `Map<String, String>` 형태로 파싱할 수 있다.
* 브라우저 개발자 도구의 Headers 탭에서는 HTTP 요청 원문이 그대로 보이지 않고 `Request URL`, `Request Method`, `Request Headers`처럼 분리되어 보인다. 실제 Request Line은 서버에서 `BufferedReader.readLine()`으로 읽어 로그로 확인하는 것이 더 정확했다.
* 한글 이름이나 이메일의 `@` 같은 문자는 URL에 그대로 실리지 않고 인코딩되어 전달된다. 예를 들어 한글은 `%ED...` 형태로, `@`는 `%40`으로 변환된다.
* GET 방식은 전달된 값이 URL에 그대로 드러나기 때문에, 동작 흐름을 이해하기에는 편하지만 비밀번호처럼 민감한 데이터를 전달하는 방식으로는 적절하지 않다는 점도 알게 되었다.
* 회원가입 요청을 처리할 때는 요청 경로가 `/user/create`인지 확인하고, Query String에서 꺼낸 값으로 `User` 객체를 생성한 뒤 `DataBase`에 저장하는 흐름으로 구현할 수 있다.

#### 궁금한 내용
* Query String으로 전달된 한글과 특수문자를 서버에서 정확히 복원하려면 어떤 디코딩 처리가 필요한지 더 확인해 보고 싶다.
* GET 방식은 URL에 데이터가 그대로 노출되는데, 실제 서비스에서는 어떤 기준으로 POST 방식으로 전환하는지 더 알아보고 싶다.
* 브라우저 개발자 도구에서 raw HTTP request message 전체를 바로 확인할 수 있는 방법이 있는지 더 찾아보고 싶다.

### 요구사항 3 - post 방식으로 회원가입
#### 새로 알게 된 내용
* GET은 서버에 이미 존재하는 데이터를 조회하는 역할에 가깝고, 일반적으로 서버의 상태를 변경하지 않는다. 반면 POST는 서버에 요청을 보내 데이터 추가, 수정, 삭제와 같이 상태를 변경하는 작업을 수행할 때 사용한다.
* 따라서 회원가입처럼 새로운 사용자를 생성하는 기능은 GET보다 POST 방식이 더 적절하다.
* `form`의 `method`를 `post`로 변경하면, 더 이상 사용자가 입력한 값이 URL 뒤의 Query String에 붙지 않고 Request Body에 담겨 전달된다.
* POST 방식으로 바뀌면 브라우저 개발자 도구에서 `/user/create?userId=...`처럼 길게 보이던 URL이 `/user/create`처럼 간단하게 보이고, 실제 사용자 입력값은 Request Payload 또는 Body 영역에서 확인할 수 있다.
* 서버에서는 Request Line의 URI만 보고 데이터를 모두 처리하는 것이 아니라, 헤더에서 `Content-Length` 값을 확인한 뒤 그 길이만큼 Request Body를 읽어야 한다.
* 이 프로젝트에서는 `IOUtils.readData()`를 사용해 `BufferedReader`에서 Body 데이터를 읽을 수 있었다.
* POST로 전송된 폼 데이터도 `userId=...&password=...&name=...&email=...` 형식의 문자열로 전달되기 때문에, GET 방식에서 사용했던 `HttpRequestUtils.parseQueryString()`을 그대로 활용해 파라미터를 파싱할 수 있다.
* GET 방식은 입력값이 URL에 그대로 노출되기 때문에 비밀번호 같은 민감한 정보가 주소창에 드러난다. POST 방식은 이런 값이 URL에 직접 노출되지 않기 때문에 회원가입 처리에 더 적절하다는 점을 알게 되었다.
* 회원가입 요청을 처리할 때는 `/user/create` 요청인지 확인하고, Body에서 읽은 값을 바탕으로 `User` 객체를 생성한 뒤 `DataBase`에 저장하는 흐름으로 구현할 수 있다.

#### 궁금한 내용
* POST 요청의 Body는 언제나 같은 형식으로 들어오는지, 아니면 `Content-Type`에 따라 읽는 방식이 달라지는지 더 알아보고 싶다.
* `Content-Length` 없이도 Body를 읽을 수 있는 경우가 있는지, 있다면 어떤 방식으로 처리하는지 궁금하다.
* GET과 POST의 차이가 단순히 데이터 위치(URL, Body)의 차이만이 아니라 HTTP 의미상 어떤 차이를 가지는지 더 정리해 보고 싶다.

### 요구사항 4 - redirect 방식으로 이동
#### 새로 알게 된 내용
* 회원가입 요청을 POST로 처리한 뒤 바로 `index.html` 파일 내용을 응답으로 내려주면, 화면은 첫 화면처럼 보일 수 있지만 브라우저 입장에서는 여전히 `POST /user/create` 요청의 응답 화면으로 인식한다.
* 이런 상태에서 새로고침을 하면 브라우저는 현재 화면을 다시 불러오기 위해 이전 POST 요청을 재전송하려고 한다. 따라서 단순히 첫 화면 HTML을 응답으로 내려주는 것만으로는 문제가 해결되지 않는다.
* 이 문제를 해결하려면 서버가 `302 Found` 응답과 `Location` 헤더를 내려 브라우저가 `/index.html`로 다시 GET 요청을 보내도록 해야 한다.
* redirect는 서버가 화면 내용을 직접 보내는 것이 아니라, 브라우저에게 다음 요청을 어디로 보낼지 알려 주는 방식이라는 점을 이해하게 되었다.
* `Location: index.html`처럼 상대 경로를 내려주면 현재 경로를 기준으로 해석되어 `/user/index.html`로 이동할 수 있다. 따라서 첫 화면으로 보내려면 `Location: /index.html`처럼 절대 경로를 사용하는 것이 안전하다.
* `response302Header(dos, "/index.html");`를 호출했다고 해서 자바 코드의 실행이 자동으로 멈추지는 않는다. redirect 응답을 보낸 뒤에도 아래 코드가 계속 실행될 수 있기 때문에 `return`이나 `else`로 흐름을 명확히 끊어 주어야 한다.
* 현재 요청에서 302 응답을 보낸 뒤 다시 200 응답과 body를 내려주려고 하면 한 요청에 두 개의 응답을 보내려는 잘못된 구조가 된다.
* 코드가 잘못되어 있어도 브라우저에서 겉으로는 정상 동작하는 것처럼 보일 수 있다는 점도 알게 되었다. 예를 들어 302 헤더가 먼저 써진 뒤 뒤쪽 코드에서 예외가 발생하면, 브라우저는 앞에서 받은 302만 해석해 `/index.html`로 이동할 수 있다. 하지만 이는 올바른 구현이 아니라 우연히 동작한 경우다.
* redirect 이후 브라우저가 다시 보내는 요청은 `GET /index.html`이므로, 그 이후 새로고침은 더 이상 회원가입 POST 요청을 재전송하지 않는다.

#### 궁금한 내용
* POST 요청 후 redirect할 때 `302`, `303`, `307` 같은 상태 코드들이 어떤 차이를 가지는지 더 알아보고 싶다.
* 브라우저가 새로고침 시 POST 재전송 경고를 띄우는 기준이 정확히 무엇인지 궁금하다.
* `Location` 헤더의 상대 경로와 절대 경로를 브라우저가 구체적으로 어떤 규칙으로 해석하는지 더 확인해 보고 싶다.

### 요구사항 5 - cookie
#### 새로 알게 된 내용
* 쿠키는 서버가 응답 헤더의 `Set-Cookie`를 통해 내려주고, 브라우저가 이를 저장한 뒤 다음 요청부터 `Cookie` 헤더에 담아 다시 보낸다.
* 즉 서버가 직접 브라우저 저장소를 만지는 것이 아니라, HTTP 응답 헤더에 `Set-Cookie`를 써서 브라우저에게 저장을 요청하는 방식이다.
* 현재 구현에서는 로그인 성공 시 `response302LoginSuccessHeader()`에서 `Set-Cookie: logined=true`를 응답 헤더에 내려주고 있다. 따라서 로그인 성공 후 브라우저가 다음 요청을 보낼 때 로그인 여부를 쿠키로 판단할 수 있다.
* 회원가입이나 로그인처럼 사용자가 입력한 값이 body에 담겨 오는 POST 요청에서는 요청 헤더에도 `Content-Length`가 존재할 수 있다. 이 값은 브라우저가 body를 먼저 만든 뒤 그 바이트 길이를 계산해서 넣어 준다.
* 평소에는 `Content-Length`를 주로 응답 헤더에서 보게 되지만, 요청에도 body가 있으면 요청 헤더에도 들어갈 수 있다는 점을 알게 되었다.
* `responseResource(out, "/user/login_failed.html")`처럼 정적 HTML 파일을 바로 응답으로 내려주면, 브라우저 주소는 그대로일 수 있지만 화면 내용은 다른 페이지처럼 보일 수 있다. 즉 URL이 바뀌는 것과 화면이 바뀌는 것은 같은 의미가 아니다.
* 반대로 redirect는 화면 내용을 직접 보내는 것이 아니라 브라우저가 새로운 URL로 다시 요청하게 만드는 방식이기 때문에, URL과 화면이 함께 바뀐다.
* 로그인 실패 시 `login_failed.html`을 바로 `200 OK`로 내려주는 것도 가능하다. 이 경우 HTTP 수준에서는 요청 처리 자체가 성공한 것이므로 `200 OK`가 될 수 있다. 다만 브라우저 입장에서는 여전히 `/user/login` 요청의 응답 화면이기 때문에 새로고침 시 이전 요청 재전송 문제가 생길 수 있다.
* 로그인 성공 후에는 쿠키를 설정하고 메인 화면으로 redirect하는 흐름이 자연스럽다. 이렇게 하면 다음 요청부터 브라우저가 쿠키를 함께 보내므로 서버가 로그인 상태를 확인할 수 있다.

#### 궁금한 내용
* `Set-Cookie`에 `Path`, `Expires`, `Max-Age`, `HttpOnly` 같은 속성을 추가하면 동작이 어떻게 달라지는지 더 알아보고 싶다.
* 브라우저가 어떤 쿠키를 어떤 경로에 대해 다시 보내는지, 특히 `Path`가 없을 때 기본 동작이 어떻게 정해지는지 궁금하다.
* 로그인 실패 화면도 redirect로 처리하는 편이 더 적절한지, 아니면 현재처럼 `200 OK`로 실패 페이지를 직접 내려주는 것이 더 나은지 상황별 기준을 더 정리해 보고 싶다.
